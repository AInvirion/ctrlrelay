# CLI reference

The `dev-sync` console script is a [Typer](https://typer.tiangolo.com/) app
defined at
[`src/dev_sync/cli.py`](https://github.com/AInvirion/dev-sync/blob/main/src/dev_sync/cli.py).
This page is generated by hand from that source — when in doubt, read the file.

Run any command with no arguments (or `--help`) to see Typer's auto-generated
help.

```text
dev-sync [--version] <command> ...
```

| Top-level | Purpose |
|---|---|
| `dev-sync --version` / `-v` | Print the package version and exit. |
| `dev-sync version` | Same, as a subcommand. |
| `dev-sync status` | Show active locks and the 5 most-recent sessions. |
| `dev-sync config ...` | Configuration helpers. |
| `dev-sync skills ...` | Skill discovery / audit. |
| `dev-sync bridge ...` | Manage the Telegram bridge daemon. |
| `dev-sync run ...` | Run a pipeline interactively. |
| `dev-sync poller ...` | Manage the issue-poller daemon. |

> **Note:** the `repos`, `export`, `import`, `team-export`, `team-import`,
> `setup`, `manifest`, `codex-export`, `codex-import`, and `codex-install`
> commands are **shell-script** subcommands of the legacy `./sync` wrapper, not
> the Python `dev-sync` CLI. See the [README](https://github.com/AInvirion/dev-sync#readme)
> for those.

Every command that reads config accepts `--config` / `-c` to point at a
non-default `orchestrator.yaml` (default: `config/orchestrator.yaml`).

## `dev-sync config`

### `config validate`

```bash
dev-sync config validate [-c PATH]
```

Loads `config/orchestrator.yaml`, validates the pydantic schema, prints the
parsed `node_id`, `timezone`, transport type, and repo count. Exits non-zero on
validation errors.

### `config repos`

```bash
dev-sync config repos [-c PATH]
```

Lists configured repositories in a table (name, local path, deploy provider).

## `dev-sync skills`

### `skills audit`

```bash
dev-sync skills audit [-p SKILLS_DIR] [-c PATH]
```

Audits each skill under `paths.skills` (or `--path`) for orchestrator
readiness — checks shape, required metadata, common pitfalls. Exits non-zero if
any skill fails.

### `skills list`

```bash
dev-sync skills list [-p SKILLS_DIR] [-c PATH]
```

Lists discovered skills as a table (name, path).

## `dev-sync bridge`

See [Telegram bridge]({{ '/bridge/' | relative_url }}) for the full setup walkthrough.

### `bridge start`

```bash
dev-sync bridge start [-c PATH] [-d|--daemon]
```

Starts the bridge listening on `transport.telegram.socket_path`. Reads the
bot token from the env var named in `transport.telegram.bot_token_env`
(default `DEV_SYNC_TELEGRAM_TOKEN`). Without `--daemon` runs in the
foreground; with `--daemon` spawns a detached process and writes a PID file
alongside the socket.

Fails if `transport.type` is not `telegram`, the token env var is unset, or a
PID file already exists for a live process.

### `bridge stop`

```bash
dev-sync bridge stop [-c PATH]
```

Reads the PID file and sends `SIGTERM`.

### `bridge status`

```bash
dev-sync bridge status [-c PATH]
```

Reports running / not-running. Detects orphan socket files (socket exists but
the process is gone).

### `bridge test`

```bash
dev-sync bridge test [-m "MESSAGE"] [-c PATH]
```

Connects to the bridge socket as a client and sends a one-off message. Useful
for confirming end-to-end delivery to your Telegram chat without waiting for a
real pipeline run.

## `dev-sync run`

### `run dev`

```bash
dev-sync run dev -i ISSUE [-r REPO] [-c PATH]
```

| Flag | Required | Default | Description |
|---|---|---|---|
| `--issue` / `-i` | yes | — | GitHub issue number. |
| `--repo` / `-r` | when more than one repo is configured | — | Limit to a single `owner/repo`. |
| `--config` / `-c` | no | `config/orchestrator.yaml` | Config path. |

Acquires the per-repo lock, creates the worktree, spawns Claude with the
issue's title/body in the prompt, and reports the result. If Claude blocks and
the Telegram transport is configured, posts the question to your chat and
waits for a reply (up to `DEFAULT_MAX_BLOCKED_ROUNDS` rounds).

### `run secops`

```bash
dev-sync run secops [-r REPO] [-c PATH]
```

Runs the secops pipeline across configured repos (or one repo with `--repo`).
Each repo's run is serialised by the per-repo lock; the overall sweep runs in
parallel across repos. Reports per-repo success/failure and exits non-zero if
any repo failed.

## `dev-sync poller`

### `poller start`

```bash
dev-sync poller start [-c PATH] [-d|--daemon] [-i SECONDS]
```

| Flag | Default | Description |
|---|---|---|
| `--interval` / `-i` | `300` | Seconds between polls. |
| `--daemon` / `-d` | off | Background mode (writes PID file). |
| `--config` / `-c` | `config/orchestrator.yaml` | Config path. |

Polls each configured repo for issues assigned to your GitHub user (resolved
via `gh api user --jq .login`). On the first run, seeds the seen-issue set
with current assignments so it does not replay your existing backlog.

For each newly assigned issue, runs `run_dev_issue(...)` (i.e. the dev pipeline)
in-process. If `transport.type: telegram` is configured and the bridge socket
exists, sends notifications: `🔔 New issue` on detection, `⏸️ Blocked` on
question, `✅ PR ready` on success, `❌ Failed` otherwise.

State is persisted to `<state_db_dir>/poller_state.json`.

### `poller stop`

```bash
dev-sync poller stop [-c PATH]
```

Sends `SIGTERM` to the daemon PID.

### `poller status`

```bash
dev-sync poller status [-c PATH]
```

Reports running / not-running.

## `dev-sync status`

```bash
dev-sync status [-c PATH]
```

Opens the SQLite state DB at `paths.state_db` and prints:

- **Active locks** — `(repo → session_id)` for each held repo lock.
- **Recent sessions** — the 5 most-recent rows from the `sessions` table
  (`id`, `pipeline`, `repo`, `status`).

`status` values: `running`, `done`, `blocked`, `failed`.

If the DB doesn't exist yet, prints a hint to run a pipeline first.

## `dev-sync version`

```bash
dev-sync version
```

Prints the package version (same as `dev-sync --version`).
