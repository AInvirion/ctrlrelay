## Example 1: Subtle Logic Bug

**Scenario:** AI generates a subscription renewal date calculator.

**The bug:** When a subscription starts January 31st and renews monthly, `setMonth(getMonth() + 1)` produces March 3rd (February has no 31st). This causes inconsistent billing dates, customer complaints, and a support nightmare.

**Without VID:** "Looks good, tests pass for Jan 15 -> Feb 15." Ships to production. Bug found 3 months later via billing complaints.

**With VID (15 min, Moderate Trust):** Edge case check asks "what happens on the 31st?" Manual test reveals March 3rd instead of February 28th. Bug found and fixed before commit.

**Principles applied:** Graduated Trust (15 min matched to financial impact), Intent Before Generation (edge case thinking from upfront spec).

**Lesson:** Edge case verification is where AI-generated code most often fails. The code looks correct and handles the happy path perfectly.

## Example 2: Provenance Prevents Regression

**Scenario:** 6 months ago, AI generated an auth middleware with try-except blocks. Today, a developer modifies it to add rate limiting.

**Without provenance:** Developer assumes they wrote it, removes try-except blocks ("simplifying"), adds rate limiting outside error handling. Result: unhandled exceptions on invalid tokens, 500 errors instead of 401s, 2AM production incident.

**With provenance:** Developer sees `# Generated by AI - 2024-06-15` comment. Thinks: "I did not write this, I should verify I understand it before modifying." Checks original intent spec, understands why try-except exists, preserves error handling while adding rate limiting. No incident.

**Principles applied:** Provenance Awareness (comment triggered caution), Understanding Over Acceptance (verified understanding before modifying).

**Lesson:** AI-generated code looks identical to human-written code six months later. Without provenance markers, developers assume they understand code they did not write.

## Example 3: Risk Miscalibration

**Scenario:** Developer needs to add account deletion endpoint. Initial assessment: "straightforward CRUD, Risk Score 8, High Trust, 5 min verification."

**The reality:**
- Impact: 5 (deletes all user data, irreversible)
- Reversibility: 5 (cannot undo)
- Exposure: 3 (individual users)
- Compliance: 8 (GDPR requires specific deletion process)

**Actual score: 39 — Minimal Trust** (not 8)

**What the naive implementation missed:**
1. No authentication/authorization (anyone can delete any account)
2. GDPR non-compliance (must anonymize, not delete)
3. Orphaned related data (posts, comments, purchases)
4. No cascading effect handling
5. No audit trail
6. No grace period / undo window
7. No user notification
8. Active subscriptions not cancelled

**With proper risk calibration:** 2+ hours verification, peer review, legal/compliance review. Proper implementation includes auth, GDPR-compliant anonymization, 30-day grace period, audit logging, subscription cancellation, and user notification.

**Principles applied:** Graduated Trust (corrected from High to Minimal), Continuous Calibration (realized initial assessment was wrong).

**Lesson:** Familiarity bias is dangerous. "I have done CRUD before" does not mean THIS delete is low-risk. Systematically assessing all four factors (especially compliance) prevents catastrophic under-verification.

## Example 4: Missing Intent

**Scenario:** Developer wants to "improve query performance for the user dashboard."

**Without intent spec:** Prompt to AI: "Optimize this query." AI adds `LIMIT 100`. Query is faster, but dashboard now only shows 100 posts total (not per user). Completely wrong behavior ships to production.

**With intent spec (2 minutes):**
```
Goal: Speed up dashboard query
Current problem: Query takes 3 seconds, loads 100,000+ rows
Requirements:
- Must show ALL posts for active users (not just first N)
- Dashboard paginates on frontend
- Query should load in <500ms
```

Now AI gets proper context. Verification reveals the real solution needs query-level pagination with LATERAL joins, not a naive LIMIT.

**Principles applied:** Intent Before Generation (2 minutes prevented deploying wrong solution), Understanding Over Acceptance (verification step revealed the real problem).

**Lesson:** AI optimizes for what you ask, not what you need. "Make it faster" without context produces faster-but-wrong code.

## Example 5: Understanding Debt

**Scenario:** Junior developer asks AI to create a caching layer with double-checked locking. Understanding level: 30%.

**Month 1:** Ships with hidden issues (does not understand locking pattern).
**Month 3:** Adds invalidation that does not clean up locks (memory leak).
**Month 6:** Adds stats counters without thread safety (race conditions).
**Month 9:** Adds cache warming that bypasses locking (race conditions on writes).
**Month 12:** Production crisis — random cache corruption, 10GB+ memory leaks, data inconsistencies. 3 days senior engineer debugging. Complete rewrite needed.

**Understanding trajectory:**
```
Month 1:  30% understanding -> Ship
Month 3:  25% understanding -> Memory leak
Month 6:  20% understanding -> Race conditions
Month 9:  15% understanding -> Bypass safety
Month 12: CRISIS -> 3 days to understand + rewrite
```

**With VID (Month 1):** Verification ritual includes "explain this code to yourself." Developer realizes they do not understand double-checked locking. Invokes Principle 3 (Understanding Over Acceptance). Spends 1 hour learning threading patterns. Builds 90% understanding. All subsequent modifications are correct.

**Cost comparison:**
- Without VID: 3 days debugging + production incidents + rewrite
- With VID: 1 hour upfront

**Principles applied:** Understanding Over Acceptance (1 hour investment prevented months of compounding problems), Continuous Calibration (each modification without understanding made the next harder).

**Lesson:** Understanding debt compounds exponentially. One hour invested early prevents days lost later.

## Common Patterns

### What VID Catches

1. **Edge cases** (Jan 31st subscription)
2. **Missing context** (why try-except existed)
3. **Risk underestimation** ("simple" delete)
4. **Wrong requirements** (what "optimize" meant)
5. **Compounding debt** (understanding gap)

### Time Investment vs. Savings

| Example | VID Time | Without VID Cost | ROI |
|---------|----------|-----------------|-----|
| Subscription bug | 15 min | Hours + customer issues | 10-50x |
| Auth regression | 2 min (reading comment) | Production incident | 100x+ |
| Account deletion | 2 hours | Compliance violation | Incalculable |
| Query optimization | 2 min | Hours debugging + rollback | 30x+ |
| Caching understanding | 1 hour | 3 days + incidents + rewrite | 24x+ |

### Anti-Pattern Signals

These phrases from developers or AI suggest elevated risk:
- "This is just a quick fix"
- "We can patch it later"
- "No one will probably use this edge case"
- "The AI seems confident"
- "It is just internal tooling"
- "Looks good, tests pass" (without edge case testing)
- "I wrote this" (when provenance says otherwise)
